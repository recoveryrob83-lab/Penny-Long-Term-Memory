## Database Architecture Discussion

Conclusion reached:

- Google Sheets and Docs are documents pretending to be databases.
- Caregiver/project data needs structured records.
- A proper database may be needed for Life OS.

Suggested database direction:

- Start with SQLite as a prototype, then migrate later to MySQL/Postgres/Supabase if needed.
- MySQL is possible, but SQLite is simpler and less brittle for a first local prototype.
- Supabase/Postgres may be better long-term if auth, web app access, APIs, and row-level security become important.
- Avoid relying on ChatGPT connectors as the only automation pipe.

Suggested core database tables:

- people
- contacts
- care_profiles
- vehicles
- insurance_policies or insurance_references
- service_providers
- interactions
- documents
- privacy_labels
- audit_log

Important GitHub rule:

- GitHub can store schema, fake seed data, README files, project architecture, and migrations.
- GitHub should not store real personal data, real care recipient profiles, policy numbers, credentials, medical identifiers, banking info, or private family notes.

Suggested `.gitignore` items:

- `*.db`
- `*.sqlite`
- `*.sqlite3`
- `.env`
- `secrets.json`
- `real_data/`
- `private_exports/`

Core principle:

Documents are for humans. Databases are for facts. GitHub is for the map. Secrets belong in a vault.
